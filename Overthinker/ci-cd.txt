## 2️⃣ WHERE is the Docker image created?

### 🔹 NOT on your laptop

### 🔹 NOT on EC2

### ✅ The Docker image is built **inside GitLab CI**

📌 GitLab provides a **CI Runner (virtual machine)**

When pipeline runs:

```yaml
image: docker:24
services:
  - docker:24-dind
```

GitLab spins up a **temporary VM** like this:

```
GitLab Runner (temporary)
├── Docker installed
├── Your source code
└── Internet access
```

Inside that runner:

```bash
docker build -t yourimage .
```

➡️ **Image exists only there (temporarily)**

## 5️⃣ HOW does EC2 know to pull new image?

This is the **MOST IMPORTANT PART** 🔥

### In `.gitlab-ci.yml` (deploy stage):

```yaml
script:
  - ssh ubuntu@EC2_IP "
      docker pull yourimage:latest &&
      docker compose -f docker-compose.prod.yml up -d
    "
```

### What actually happens:

1. GitLab finishes building image
2. GitLab opens **SSH connection** to EC2
3. GitLab runs commands **ON EC2**
4. EC2 executes:

   ```bash
   docker pull yourimage:latest
   docker compose up -d
   ```
5. Old container stopped
6. New container started

⚠️ EC2 does NOT “listen”
⚠️ EC2 does NOT poll GitLab

👉 **GitLab PUSHES the update**
